Problem Statement

What We Solve

  • LLMs answer data questions from memory, not from the actual dataset
  • Hallucinated column names, wrong aggregates, unverifiable numbers
  • No reproducible audit trail for answers

Why English → Data is Hard

  • Schema is unknown at inference time
  • Multi-step reasoning across large CSVs requires iteration
  • SQL must be correct and safe — one bad query crashes trust

What MCP Adds

Grounded answers

Tool output replaces LLM memory — every number is verified

Schema inspection

Agent reads real column names before writing any SQL

Reproducibility

Every query logged to db:/query-history for audit

K

Agent Loop

Loop is mechanical — tool outputs drive correctness, not model intuition

Tools Overview

ToolPurposeReturn shape
load_csv Load a CSV file into a named SQLite table {table, row_count, columns}
list_tables List all tables currently in the in-memory DB {tables: [...]}
describe_schema Return column names and types for a table {table, columns: [{name, type}]}
run_query Execute a safe read-only SQL SELECT and return rows {columns, rows, row_count}
get_statistics Compute min, max, mean, nulls for a numeric column {column, min, max, mean, null_count}
All tools return JSON No side effects on read tools Resources: db:/schema   db:/query-history

Safety & Reliability

SQL Guard

  • run_query blocks DROP, DELETE, ALTER, INSERT, UPDATE
  • Semicolons blocked — prevents multi-statement injection
  • Only SELECT statements reach the database

Error Handling

  • All tools return {"error": "..."} on failure — no exceptions escape
  • Agent receives a valid JSON response and can retry
  • In-memory DB resets on restart — clean state every run

CSV Encoding Fallback

Tried in order until success:

1utf-8-sig
2utf-8
3cp1252 (Windows)
4latin-1
5utf-8 errors=replace (last resort)

Prompting Strategy Comparison

Same question: "Top 5 movies by revenue with vote_count ≥ 1000"

Strategy 1 — Minimal Prompt

Ask the question directly, no workflow guidance

  • Called load_csv then immediately run_query
  • Guessed movie_title — no such column → error
  • Self-repaired to title on second run_query

Strategy 3 — Expert Workflow

Schema reconnaissance before any SQL

  • Called load_csv → describe_schema → run_query
  • Read real column names first — no guessing
  • Correct SQL on first attempt, zero errors

Retro & Closing

One Improvement

Stronger SQL safety

Keyword blocking can be bypassed with obfuscated SQL. Replace with a proper AST parser (sqlglot / sqlparse) that only allows SELECT statement nodes.

Three Keeps

  • ✓ stdio transport — zero-config, no network setup
  • ✓ In-memory DB — stateless, clean per run
  • ✓ Structured JSON errors — agent can always recover
🎯

Key Takeaways

  • Grounded tool outputs beat LLM memory for data tasks
  • Schema-first prompting eliminates column hallucination
  • Safety must be architectural, not just a naming convention
